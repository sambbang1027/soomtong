# 숨통 (Soomtong) — Product Requirements Document

> 숨 막히는 지옥철, 당신의 출퇴근길에 틔워주는 단 하나의 숨통

---

## 1. 개요

### 서비스 정의
30분 단위 혼잡도 통계 데이터와 역사 구조 기반 칸별 가중치 알고리즘을 결합하여, 가장 한산한 칸을 1초 만에 안내하는 수도권 출퇴근러 맞춤형 단일 화면 유틸리티 앱.

### 핵심 가치 제안
- 역 감지 → 혼잡도 확인까지 **1초 컷**
- 10개 칸 중 **"어디에 줄 설지"**를 시스템이 직접 짚어줌
- 다른 기능 없음. 오직 이것만.

### 타겟 유저
| 구분 | 설명 |
|------|------|
| 주 타겟 | 매일 1~8호선 지옥철을 탑승하는 직장인·취준생 |
| 보조 타겟 | 밀집 공포증·공황장애로 만원 지하철을 극도로 꺼리는 교통 약자 |

## 2. 문제 정의

### 기존 서비스 한계
- 네이버 지도, 카카오맵, TMAP은 **시간(언제 오나)**과 **경로(어떻게 가나)**에만 집중
- 칸별 혼잡도 기능이 있어도 메뉴 깊숙이 숨겨져 있어 승강장에서 즉시 확인 불가

### 숨통의 차별화
- 단 하나의 질문 — **"몇 번 칸이 제일 널널한가"** — 에만 답함
- 승강장 도착부터 칸 결정까지 3초 이내 UX

### 데이터 환경의 한계와 설계 판단
숨통 개발 과정에서 다음과 같은 데이터 제약을 확인했고, 각각에 대해 명시적인 설계 판단을 내렸다.

| 검토한 데이터 | 한계 | 판단 |
|---|---|---|
| SKT 지오비전 퍼즐 칸별 혼잡도 API | 유료 B2B 전환, 개인 접근 불가 | ❌ 제외 |
| 서울시 실시간 도시데이터 (OA-21285) | 122개 장소 한정, 칸별 데이터 없음 | ❌ 제외 |
| 서울시 시간대별 승하차 인원 (OA-12252) | 역 전체 단위 인원수, 칸별 분리 없음, 전월 데이터 | ❌ 제외 |
| 9호선 혼잡도 데이터 | 2020~2022년 자료만 존재, 최신 데이터 없음 | ❌ MVP 제외 |
| 서울교통공사 혼잡도 (1~8호선) | 열차 전체 단위 %, 파일이 아닌 API 제공 | ✅ 채택 |
| 서울 열린데이터광장 실시간 도착정보 | 실시간 열차 도착 시간 제공 | ✅ 채택 |

**칸별 혼잡도 추정 방식 채택 이유:**
공개된 데이터 중 칸별로 분리된 데이터는 존재하지 않는다. 이에 역사 구조(계단·에스컬레이터·환승 통로·출구 위치)를 직접 분석하여 칸별 가중치 테이블을 설계하고, 전체 혼잡도에 곱하는 방식으로 추정치를 산출한다.

**정확도에 대한 솔직한 인식:**
- 이 방식은 "방향성(어느 칸이 상대적으로 여유로운가)"은 맞을 가능성이 높으나, 구체적인 % 수치는 신뢰하기 어렵다.
- 따라서 UI에서 % 숫자를 직접 노출하지 않고, 색상(초록/노랑/빨강)과 텍스트(여유/보통/혼잡)만 표시하는 UX 결정을 내렸다.
- "부정확한 수치를 숨기는 것"이 아니라, "신뢰할 수 있는 정보만 노출하는 것"이 설계 원칙이다.

## 3. 기능 요구사항

### 3.1 Feature 1 — GPS 기반 현재 역 자동 감지
- 앱 진입 시 브라우저 Geolocation API로 현재 위치 획득
- 수도권 지하철역 좌표 DB와 비교해 가장 가까운 역·호선 자동 선택
- 위치 권한 거부 시 수동 역 검색 폴백(fallback) 제공

### 3.2 Feature 2 — 칸별 혼잡도 시각화 (Subway Grid View)
- 열차 모양의 가로형 블록 10개 렌더링
- 전체 혼잡도 × 역사 구조 가중치로 산출한 추정 혼잡도 기반 3단계 색상 전환
- % 수치는 노출하지 않음. 색상과 텍스트 레이블만 표시

| 단계 | 기준 | 색상 | 레이블 |
|------|------|------|--------|
| 여유 | 추정 혼잡도 30% 이하 | 🟢 민트/에메랄드 | 여유 |
| 보통 | 추정 혼잡도 31–70% | 🟡 옐로우 | 보통 |
| 혼잡 | 추정 혼잡도 71% 이상 | 🔴 레드 | 혼잡 |

### 3.3 Feature 3 — 숨통 칸 추천 (Best Pick)
- `Math.min`으로 최저 추정 혼잡도 칸 인덱스 자동 계산
- 화면 상단 추천 배너에 큰 글씨로 표시
- 예: *"지금은 9번 칸이 가장 숨통 트입니다!"*
- % 수치 미노출 (색상 기반 판단 유도)

### 3.4 Feature 4 — 실시간 열차 도착 정보
- 서울 열린데이터광장 OA-12764로 다음 열차 도착까지 남은 시간 표시
- 예: *"2분 후 도착"*
- 유저가 추천 칸으로 이동할 시간이 있는지 체감할 수 있게 함

### 3.5 Feature 5 — 상행/하행 방향 선택
- 화면 상단에 상행/하행 토글 버튼 배치
- 선택 시 페이지 이동 없이 열차 그리드와 추천 배너만 해당 방향 데이터로 즉시 교체
- 기본값: 현재 시간대 기준 혼잡한 방향으로 자동 선택 (판단 어려울 경우 상행 기본)

### 3.6 Feature 6 — 역사 구조 팁
- 화면 하단에 현재 역의 가중치 산출 근거를 자연어로 표시
- 예: *"강남역은 3·4번 칸 계단이 몰려 있어 9번 칸이 상대적으로 여유롭습니다"*
- `stationWeights.js`의 `tip` 필드에서 가져옴
- 가중치 데이터가 없는 역(균등 가중치 적용 역)은 팁 영역 미표시

## 4. 데이터 및 알고리즘 설계

### 4.1 사용 API

| 역할 | API | 출처 | 형태 |
|---|---|---|---|
| 전체 혼잡도 % | 서울교통공사 지하철혼잡도정보 | 공공데이터포털 | API (JSON) |
| 실시간 열차 도착 | 지하철 실시간 도착정보 OA-12764 | 서울 열린데이터광장 | API (JSON) |

### 4.2 혼잡도 API 응답 구조
```json
{
  "data": [
    {
      "요일구분": "평일",
      "호선": "2",
      "역번호": 222,
      "출발역": "강남",
      "상하구분": "상행",
      "18시00분": "68",
      "18시30분": "74"
    }
  ]
}
```

### 4.3 현재 시간대 슬롯 추출
```js
const now = new Date()
const hour = now.getHours()
const min = now.getMinutes() < 30 ? '00' : '30'
const key = `${hour}시${min}분` // → "18시00분"
const congestion = Number(data[key])  // string → number 변환 필요
```

### 4.4 칸별 가중치 테이블 (stationWeights.js)
역사 구조(계단·에스컬레이터·환승 통로·출구 위치) 직접 조사 기반.
또타 지하철 앱 "빠른 환승" 기능 + 서울교통공사 역사 안내도 참고.
가중치 합은 10.0으로 고정하여 전체 혼잡도 비율 유지.

```js
export const STATION_WEIGHTS = {
  "강남": {
    "상행": [0.8, 0.9, 1.4, 1.5, 1.2, 1.0, 0.9, 0.8, 0.7, 0.7],
    "하행": [0.7, 0.8, 1.3, 1.4, 1.1, 1.0, 0.9, 0.8, 0.8, 0.9],
  },
  // 주요 역 10~15개 수작업 조사
}
```

**최종 공식:**
```
칸별 추정 혼잡도(%) = 전체 혼잡도(%) × 해당 칸 가중치
```

### 4.5 가중치 설계 원칙
- **현재 역 계단 위치** → 승차 집중 칸 가중치 높임
- **현재 역 환승 통로 위치** → 환승 집중 칸 가중치 높임
- **현재 역 주요 출구 위치** → 하차 집중 칸 가중치 높임

### 4.6 커버리지 미달 역 폴백 정책
가중치 테이블 MVP 범위는 주요 역 10~15개로 한정한다. 테이블에 없는 역은 모든 칸에 가중치 1.0(균등)을 적용한다.
- 칸별 색상·추천은 전체 혼잡도만 기준으로 동작 (방향성 안내 불가)
- 역사 구조 팁 미표시
- 배포 후 역 추가는 `stationWeights.js`에 데이터만 추가하면 적용됨

> **알려진 한계:** 이전 정거장의 계단 위치에 따라 하차 후 빈 칸이 생길 수 있으나, MVP에서는 현재 역 기준 가중치만 적용한다. 이전 역 하차 패턴 반영은 v2 고도화 과제로 명시한다.

## 6. 기술 스택

| 구분 | 선택 | 이유 |
|------|------|------|
| 프레임워크 | React + Vite | 빠른 HMR, 경량 번들 |
| 스타일 | Tailwind CSS | 유틸리티 클래스로 빠른 UI 구성 |
| HTTP 클라이언트 | fetch | 외부 라이브러리 없이 충분 |
| 혼잡도 데이터 | 서울교통공사 혼잡도 API (공공데이터포털) | 무료, 30분 단위 % 제공 |
| 도착 정보 | 서울 열린데이터광장 OA-12764 | 무료, 실시간 도착 시간 제공 |
| 배포 | Vercel | GitHub 연동 자동 배포, rewrites 지원 |
| 백엔드 | 없음 | 순수 프론트엔드, Vite 프록시로 CORS 처리 |

### API 연동 구조
```
[Browser]
    │ GET /api/congestion/...
    ▼
[Vite Proxy (dev) / Vercel Rewrite (prod)]
    │ GET https://api.data.go.kr/...
    ▼
[공공데이터포털 API]
```

## 8. 개발 일정 (3일 스프린트)

### Day 1 — 데이터 통신 뼈대
**목표:** 콘솔에 혼잡도 숫자와 도착 정보가 에러 없이 출력

- [ ] Vite + React + Tailwind 환경 세팅
- [ ] 공공데이터포털 API 키 발급
- [ ] 서울 열린데이터광장 API 키 발급
- [ ] `vite.config.js` 프록시 설정 (CORS 해결)
- [ ] 혼잡도 API 호출 및 응답 구조 검증
- [ ] 도착 정보 API 호출 및 응답 구조 검증
- [ ] **완료 기준:** 역 이름 입력 → 콘솔에 `{ congestion: 68, arrival: "2분 후" }` 출력

### Day 2 — 핵심 UI + 알고리즘 연동
**목표:** 역 선택 시 10개 칸 색상이 바뀌고, 추천 칸이 텍스트로 표시됨

- [ ] `stationWeights.js` 주요 역 가중치 테이블 작성 (10~15개 역)
- [ ] `calcCongestion.js` 계산 로직 구현
- [ ] `SubwayGrid.jsx` 그리드 마크업 + 색상 조건부 스타일링
- [ ] `RecommendBanner.jsx` Math.min 기반 최적 칸 추출 + 배너 출력
- [ ] `ArrivalInfo.jsx` 도착 정보 연동
- [ ] **완료 기준:** 역 검색 → 10개 칸 색깔 변하고 추천 칸 텍스트 출력

### Day 3 — 디자인 폴리싱 + 배포
**목표:** 퇴근길 승강장에서 내 폰으로 실제 사용 가능한 앱

- [ ] GPS 자동 감지 (`useGeolocation` 훅) 연동
- [ ] 모바일 반응형 레이아웃 최적화 (max-width 430px)
- [ ] 호선별 브랜드 컬러 포인트 적용
- [ ] GitHub 업로드 → Vercel 배포
- [ ] **완료 기준:** 승강장에서 내가 만든 앱으로 칸 골라서 탑승

## 10. 향후 확장 아이디어 (Post-MVP)

- **PWA 홈 화면 추가:** iOS Safari / Android Chrome 홈 화면 설치 지원
